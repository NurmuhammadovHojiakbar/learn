# Secrets & Environments

## What you'll learn
- Storing credentials as **encrypted secrets**
- The difference between **secrets** and **variables**
- **Environments** with protection rules and approvals
- **OIDC** — short-lived cloud credentials instead of stored keys

Never put a token, password, or API key in a workflow file — it's in your git history
forever and visible to anyone with read access. Use the mechanisms below.

---

## Secrets vs variables
| | Secret | Variable |
|--|--------|----------|
| Encrypted | ✅ yes | ❌ no |
| Masked in logs | ✅ yes | ❌ no |
| Use for | tokens, passwords, keys | non-sensitive config (region, URL) |
| Accessed via | `${{ secrets.NAME }}` | `${{ vars.NAME }}` |

Set both in **Repo → Settings → Secrets and variables → Actions** (or at the
organization/environment level).

```yaml
steps:
  - name: Deploy
    run: ./deploy.sh
    env:
      API_TOKEN: ${{ secrets.API_TOKEN }}     # encrypted, masked in logs
      AWS_REGION: ${{ vars.AWS_REGION }}      # plain config
```

GitHub automatically **masks** secret values in logs (they appear as `***`). Don't defeat
this by `echo`-ing them or writing them to files that get uploaded as artifacts.

---

## Scopes
Secrets/variables can be defined at three levels (more specific wins):
1. **Repository** — available to all workflows in the repo.
2. **Environment** — only to jobs targeting that environment (see below).
3. **Organization** — shared across many repos (with access policies).

`GITHUB_TOKEN` is a **built-in**, auto-generated secret for authenticating to the GitHub API
(e.g. pushing to GHCR, commenting on PRs). Scope its permissions with `permissions:`:
```yaml
permissions:
  contents: read
  packages: write        # e.g. to push images to GHCR
```

---

## Environments
An **environment** (e.g. `staging`, `production`) is a named deployment target with its own
secrets and, crucially, **protection rules**.

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production       # ← ties this job to the "production" environment
    steps:
      - run: ./deploy.sh
        env:
          DEPLOY_KEY: ${{ secrets.DEPLOY_KEY }}   # this is the PRODUCTION secret
```

Configure environments in **Settings → Environments**. Protection rules include:
- **Required reviewers** — the deploy job **pauses** until an approver clicks "Approve"
  (this is how you get Continuous *Delivery* with a manual gate).
- **Wait timer** — delay before deploying.
- **Deployment branches** — restrict which branches can deploy (e.g. only `main`).

So the same workflow can deploy to `staging` automatically but require approval for
`production`, just by targeting different environments.

## OIDC — stop storing cloud keys
Long-lived cloud keys (`AWS_SECRET_ACCESS_KEY`, etc.) stored as secrets are a liability —
they don't rotate and leak badly. **OpenID Connect (OIDC)** lets a workflow request a
**short-lived token** directly from your cloud provider, with no stored credentials.

```yaml
permissions:
  id-token: write        # required for the workflow to request an OIDC token
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/github-deploy
          aws-region: us-east-1
      - run: aws s3 sync ./dist s3://my-bucket   # authenticated via short-lived creds
```
You configure a trust relationship in the cloud provider once; thereafter no secret keys
live in GitHub at all. AWS, GCP, and Azure all support this.

## Common pitfalls
- ❌ Hardcoding tokens in YAML or committing `.env` files.
- ❌ `echo "${{ secrets.X }}"` or writing secrets to uploaded artifacts → leak past masking.
- ❌ Over-broad `GITHUB_TOKEN` permissions → use least privilege with `permissions:`.
- ❌ Long-lived cloud keys when OIDC is available.
- ❌ Forgetting `environment:` on the deploy job, so approval gates never apply.

## Best practices
- ✅ Secrets for sensitive values, variables for config — never inline.
- ✅ Use **environments** with required reviewers to gate production.
- ✅ Scope `GITHUB_TOKEN` with explicit `permissions:` (least privilege).
- ✅ Prefer **OIDC** over stored cloud credentials.
- ✅ Scope secrets to environments so prod creds aren't exposed to non-prod jobs.

## Try it yourself
1. Add a repo secret `GREETING_SECRET` and a step that uses it via `env:` — confirm it's
   masked as `***` in the logs.
2. Create a `production` environment with yourself as a required reviewer; add a job with
   `environment: production` and watch the run **pause** for approval.
3. (Cloud) Set up OIDC for AWS/GCP/Azure and assume a role with no stored keys.
