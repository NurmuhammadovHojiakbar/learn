## Why Compose?
Running a real app means juggling several containers â€” your API, a database, Redis, maybe a
worker â€” each with its own ports, volumes, network, and env. Doing that with long
`docker run` commands is painful and error-prone.

**Docker Compose** declares the whole stack in one file and brings it up/down with one
command. It automatically creates a shared network so services find each other **by name**.

## A Node + Redis stack
Create `compose.yaml` (also accepted: `docker-compose.yml`):

```yaml
services:
  api:
    build: .                     # build from the Dockerfile in this directory
    ports:
      - "3000:3000"              # host:container
    environment:
      REDIS_URL: redis://redis:6379   # "redis" = the service name below
      NODE_ENV: production
    depends_on:
      redis:
        condition: service_healthy    # wait until Redis passes its healthcheck
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    volumes:
      - redis-data:/data         # persist Redis data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5

volumes:
  redis-data:                    # named volume, managed by Docker
```

Bring it up:
```bash
docker compose up -d           # build (if needed) + start in the background
docker compose ps              # see services
docker compose logs -f api     # tail one service's logs
curl localhost:3000
docker compose down            # stop & remove containers + network
docker compose down -v         # also remove named volumes (wipes data)
```

> The Node app reaches Redis at `redis://redis:6379` â€” the **service name** is the hostname,
> thanks to Compose's automatic network. This is the [Lesson 05](./05-volumes-and-networking.mdx)
> networking concept, done for you.

## Key fields explained
| Field | Purpose |
|-------|---------|
| `build` / `image` | Build from a Dockerfile, or pull a prebuilt image |
| `ports` | Publish container ports to the host |
| `environment` / `env_file` | Inject config (inline, or from a `.env` file) |
| `volumes` | Mount named volumes or bind mounts |
| `depends_on` | Start ordering; with `condition: service_healthy`, wait for health |
| `healthcheck` | How Compose decides a service is ready |
| `restart` | `no` / `on-failure` / `always` / `unless-stopped` |
| `networks` | Custom networks (a default one is created automatically) |

> **`depends_on` caveat:** without a healthcheck condition, it only waits for the container
> to *start*, not to be *ready*. Use `condition: service_healthy` (as above) so your API
> doesn't race ahead of Redis.

## Dev vs prod with overrides
Compose merges `compose.yaml` with an optional `compose.override.yaml` automatically. Keep
production config in the base file and dev-only tweaks (bind mounts, hot reload) in the
override:

```yaml
# compose.override.yaml (dev only)
services:
  api:
    build:
      target: builder            # use the dev stage with dev deps
    volumes:
      - .:/app                    # live code editing
      - /app/node_modules         # keep container's node_modules
    command: ["npm", "run", "dev"]
    environment:
      NODE_ENV: development
```

For production you'd run `docker compose -f compose.yaml up -d` (base only).

## Handy commands
```bash
docker compose up -d --build      # rebuild images then start
docker compose exec api sh        # shell into the running api service
docker compose restart api
docker compose stop               # stop without removing
docker compose down               # stop + remove containers/network
docker compose config             # validate & print the resolved config
```

- âś… One service per container; reference others by service name.

- âś… Persist stateful services with named volumes.
