# Introduction to Docker

## What you'll learn
- What a container actually is, and how it differs from a VM

- Installing Docker and running your first container
- The handful of commands you'll use every day

## What is Docker?
**Docker** packages an application together with everything it needs to run — code,
runtime, libraries, system tools — into a single, portable unit called a **container**.
A container runs the same way on your laptop, a teammate's machine, CI, and production.

It's the cure for *"but it works on my machine."*

## Containers vs Virtual Machines
Both isolate workloads, but at different levels:

```
   Virtual Machines                 Containers
 ┌───────┬───────┬───────┐       ┌───────┬───────┬───────┐
 │ App A │ App B │ App C │       │ App A │ App B │ App C │
 ├───────┼───────┼───────┤       ├───────┴───────┴───────┤
 │Guest OS│Guest OS│Guest OS│      │   Docker Engine       │
 ├───────┴───────┴───────┤       ├───────────────────────┤
 │      Hypervisor       │       │       Host OS         │
 ├───────────────────────┤       ├───────────────────────┤
 │       Host OS         │       │      Hardware         │
 └───────────────────────┘       └───────────────────────┘
```

| | Virtual Machine | Container |
|--|-----------------|-----------|
| Isolates at | Hardware (full guest OS) | Process (shares host kernel) |
| Size | Gigabytes | Megabytes |
| Startup | Minutes | Milliseconds–seconds |
| Overhead | High | Low |

Containers share the host kernel instead of booting a whole OS, so they're **far lighter
and faster** — you can run many on one machine.

## Why developers use Docker
- **Consistency** — identical environment everywhere, from dev to prod.
- **Isolation** — run Postgres 14 and Redis 7 for one project without polluting your system.
- **Fast onboarding** — `docker compose up` instead of a page of setup instructions.
- **Portability** — the same image runs on any host with Docker.
- **CI/CD ready** — build an image once, test it, ship the exact same artifact.

## Install & verify
- **Mac/Windows:** Docker Desktop. **Linux:** Docker Engine (`curl -fsSL https://get.docker.com | sh`).

```bash
docker --version
docker run hello-world
```
`hello-world` pulls a tiny image from Docker Hub and runs it. If you see the welcome
message, Docker works.

## Core vocabulary
- **Image** — a read-only template (your app + dependencies). Like a class.
- **Container** — a running instance of an image. Like an object.
- **Dockerfile** — the recipe used to build an image.
- **Registry** — where images live (Docker Hub, GHCR). Like npm for images.
- **Volume** — persistent storage that outlives a container.

## Your first real container
```bash
# Run nginx in the background, mapping host port 8080 -> container port 80
docker run -d -p 8080:80 --name web nginx

docker ps                 # see it running
curl localhost:8080       # the nginx welcome page
docker logs web           # view its logs
docker stop web           # stop it
docker rm web             # remove it
```

The `-p host:container` flag is how the outside world reaches a process inside the
container. Without it, the container's ports stay private.

## Common pitfalls
- ❌ Forgetting `-p` and wondering why you can't reach the app.
- ❌ Thinking containers persist data by default — they don't (see [Lesson 05](./05-volumes-and-networking.mdx)).
- ❌ Confusing images (templates) with containers (running instances).
- ❌ Running everything as root inside the container (we fix this in [Lesson 04](./04-dockerize-node-app.mdx)).

## Best practices
- ✅ One main process per container (a web server *or* a database, not both).
- ✅ Treat containers as **disposable** — recreate, don't hand-patch them.
- ✅ Keep state in volumes or external services, not inside the container.

## Try it yourself
1. Run `docker run hello-world` and read the output.
2. Start nginx as above, open http://localhost:8080, then stop and remove it.
3. Run `docker run -it ubuntu bash` to drop into an interactive Ubuntu shell, run `ls /`,
   then `exit`. Notice how fast it started compared to a VM.
