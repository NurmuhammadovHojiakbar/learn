# Writing a Dockerfile

## What you'll learn
- The essential Dockerfile instructions
- How **layer caching** works and how to order instructions to exploit it
- `COPY` vs `ADD`, `CMD` vs `ENTRYPOINT`
- How to build and tag your image

A **Dockerfile** is the recipe for an image. `docker build` reads it top to bottom, running
each instruction and producing a cached layer.

---

## A first Dockerfile
```dockerfile
# Start from a base image (the foundation layer)
FROM node:20-alpine

# Set the working directory inside the image
WORKDIR /app

# Copy dependency manifests first (see caching note below)
COPY package*.json ./

# Install dependencies
RUN npm install

# Copy the rest of the source
COPY . .

# Document the port the app listens on
EXPOSE 3000

# The command that runs when a container starts
CMD ["node", "app.js"]
```

Build and run it:
```bash
docker build -t myapp .          # "." = build context (this directory)
docker run -p 3000:3000 myapp
```

---

## The essential instructions
| Instruction | Purpose |
|-------------|---------|
| `FROM` | Base image to build on |
| `WORKDIR` | Set/create the working directory |
| `COPY` | Copy files from the build context into the image |
| `ADD` | Like COPY, plus URL download & tar auto-extract |
| `RUN` | Execute a command at **build** time (creates a layer) |
| `ENV` | Set an environment variable |
| `ARG` | Build-time variable (not present at runtime) |
| `EXPOSE` | Document a port (doesn't publish it ‚Äî `-p` does) |
| `CMD` | Default command at **run** time |
| `ENTRYPOINT` | The executable; CMD becomes its args |
| `USER` | Switch to a non-root user |
| `HEALTHCHECK` | How Docker tests the container is healthy |

## Layer caching ‚Äî the most important concept
Each instruction produces a cached layer. On rebuild, Docker reuses cached layers **until**
it hits an instruction whose inputs changed ‚Äî then it rebuilds that layer and everything
after it.

**Consequence:** order instructions from *least* to *most* frequently changing. Your source
code changes constantly; your dependencies rarely. So copy and install dependencies
**before** copying source:

```dockerfile
COPY package*.json ./     # changes rarely
RUN npm install           # ‚Üê stays cached as long as package*.json is unchanged
COPY . .                  # changes every commit (only this layer rebuilds)
```

If you instead did `COPY . .` before `npm install`, every code change would bust the cache
and reinstall all dependencies ‚Äî slow builds. This ordering trick saves minutes per build.

---

## COPY vs ADD
- **`COPY`** ‚Äî straightforward copy from build context. **Prefer this.**
- **`ADD`** ‚Äî also fetches remote URLs and auto-extracts local tar archives. The magic is
  surprising; only use it when you specifically need extraction.

```dockerfile
COPY ./src /app/src              # ‚úÖ clear
ADD https://example.com/x.tar /  # ‚öÝÔ∏è only when you need ADD's extras
```

## CMD vs ENTRYPOINT
Both define what runs when the container starts, but differ in intent:

- **`CMD`** ‚Äî the *default* command; easily overridden at `docker run`.
- **`ENTRYPOINT`** ‚Äî the fixed executable; `CMD` (or run args) become its arguments.

```dockerfile
# CMD only ‚Äî override by passing a command:
CMD ["node", "app.js"]
# docker run myapp            -> node app.js
# docker run myapp npm test   -> npm test (CMD replaced)

# ENTRYPOINT + CMD ‚Äî fixed program, default args:
ENTRYPOINT ["node"]
CMD ["app.js"]
# docker run myapp            -> node app.js
# docker run myapp server.js  -> node server.js (only the arg changed)
```

Use the **exec form** (`["node", "app.js"]`, a JSON array) rather than the shell form
(`node app.js`) so signals like `SIGTERM` reach your process for graceful shutdown.

## ENV and ARG
```dockerfile
ARG NODE_ENV=production      # available only during build
ENV NODE_ENV=$NODE_ENV       # baked into the image, present at runtime
ENV PORT=3000
```
Pass build args with `docker build --build-arg NODE_ENV=development .`.

## Common pitfalls
- ‚ùå `COPY . .` before installing deps ‚Üí cache busts on every code change.
- ‚ùå Shell-form `CMD node app.js` ‚Üí signals don't reach Node; slow/abrupt shutdowns.
- ‚ùå Using `ADD` for plain copies ‚Üí surprising behavior; use `COPY`.
- ‚ùå Forgetting a `.dockerignore` ‚Üí `node_modules` and secrets bloat the image
  (see [Lesson 04](./04-dockerize-node-app.mdx)).
- ‚ùå Putting secrets in `ENV`/`ARG` ‚Üí they're baked into image layers and visible.

## Best practices
- ‚úÖ Order instructions least‚Üímost frequently changing for cache hits.
- ‚úÖ Prefer `COPY`; use exec-form `CMD`/`ENTRYPOINT`.
- ‚úÖ Pin a specific base tag (`node:20-alpine`).
- ‚úÖ Combine related `RUN` steps with `&&` to reduce layers, and clean up in the same layer.

## Try it yourself
1. Write the Dockerfile above for a tiny `app.js` that prints "hello from docker".
2. Build it, then change `app.js` and rebuild ‚Äî watch which layers are `CACHED`.
3. Now move `COPY . .` above `RUN npm install`, rebuild after a code change, and observe
   that dependencies reinstall. Put it back the right way.
